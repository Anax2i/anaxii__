## 内容对等

从语法和句子结构上来说，翻译必须具有与原文相同的信息，这意味着翻译必须与英语文章严格地在内容上相等。对译文任何更改都必须与英文文章的含义相同。

但在某些情况下，允许内容有所不同：

- 最初以英语以外的其他语言撰写的文章（在这种情况下，英语文章被视为译文）。
- 对在 osu! 社区中常见的英语单词的解释。
- 外部链接
- 标签
- 小团体中特殊的解释

## 前言

前言（也即元数据）必须放置在文件的最顶部。它由 [YAML](https://baike.baidu.com/item/YAML/1067697) 编写，并描述了有关该文章的其他信息。元数据必须在其上下两行用三个连字符（`---`）包围，并且在其后必须有一空行。

### 需要帮助整理的文章

*注：避免在翻译英文文章时使用此标签，但文章确实需要整理的时候亦需使用。*

将 `needs_cleanup` 标记添加到需要重写或格式化帮助的文章中。为此，也可以在 GitHub 上发布议题（issue）。该标签必须遵守下列写法：

```yaml
needs_cleanup: true
```

在添加此标签时，应添加相应的[注释](#注释)说明需要进行怎样的更新才能移除该标签。

### 过期文章

*注：翻译文章时不应使用此标签，如果英文文章使用了此标签意味着译文也必须使用此标签。*

当英文文章更新时，过时的翻译文章必须使用 `outdated` 标记，当英文文章包含的内容具有误导性或不再相关时，它们也可能会过期。该标签必须遵守下列写法：

```yaml
outdated: true
```

在添加此标签时，应添加相应的[注释](#注释)说明需要进行怎样的处理才能移除该标签。

### 给文章添加标签

标签可帮助网站的搜索引擎更好地寻找文章，应使用与文章相同的语言编写标签，同时附上原本的标签列表。可以的情况下使用小写字母。

例如，一篇名为“谱面讨论”的文章可能包含以下标记：

```yaml
tags:
  - beatmap discussions
  - modding V2
  - MV2
  - 谱面讨论
  - 摸图系统第二代
```

### 没有校对的翻译

*注：Wiki 维护者将在合并时决定是否使用该标签。*

有时，一篇翻译文章因为各种原因没有经过母语校对而被添加到 wiki 中，在这种情况下，将添加 `no_native_review` 标记，以使将来的翻译者知道可能需要对译文进行检查。该标签必须遵守下列写法：

```yaml
no_native_review: true
```

## 文章命名

*译注：本小节内容是针对英文文章的*
*另见：[文件夹名](#文件夹名)和[标题](#标题)*

文章标题应该使用单数形式，句首大写，可参阅[维基百科的 Naming conventions 文章（英文）](https://en.wikipedia.org/wiki/Wikipedia:Naming_conventions_(plurals))。

文章标题应该与其位于的文件夹名匹配（空格尽可能替换为下划线），文件夹名改动时，文章标题也需改动，反之亦然。

---

比赛文章是例外，文件夹名必须为缩写或首字母略缩词，而文章标题必须为比赛的完整名称。

## 文件夹与文件结构

### 文件夹名

*另见：[文章命名](#文章命名)*

文件夹名必须是英文，句首大写。

文件夹名必须且只能使用以下字符：

- 大小写字符
- 数字
- 下划线（`_`）
- 连字符（`-`）
- 感叹号（`!`）

### 文件名

文章的文件名可在[本地化小节]的 `文件名` 列找到。译文与英文原文位于同一文件夹内。

### 索引文章

即使某个文件夹是为了存放其他文章，也必须建立一篇索引文章。索引文章需列出文件夹下有哪些文章，也可以包含其他信息，如导语或这些文章的总括。

### 消歧义文章

[消歧义](/wiki/Disambiguation)文章必须放在 `/wiki/Disambiguation` 目录下。主页必须进行更新包含消歧义的文章。比如：[Disambiguation/Mod](/wiki/Disambiguation/Mod)

重定向必须进行更新，以使消歧义关键字重定向到消歧义文章。

## HTML

HTML 禁止使用，除非是用于[注释](#注释)。使用 HTML 进行排版意味着文章需要重构。

### 注释

HTML 注释应当用于记录待办事项，也可以用于注解。注释应放在单独的一行，但也可以在段落中内联使用。内联使用时前后均不得使用空格。
*译注：对于英文文章而言是前方不得使用空格，该规则的目的是保证句子中不出现额外的空白。*

错误样例：

```markdown
HTML 注释 <!-- TODO 解释 HTML 注释 --> 应当用于记录待办事项，也可以用于注解。
```

正确样例：
```markdown
HTML 注释<!-- TODO 解释 HTML 注释 -->应当用于记录待办事项，也可以用于注解。
```

## 编辑

### 行尾序列

*注意：在 GitHub 中上传以 `CRLF` 为行尾的 Markdown 文件将导致这些文件使用 `CRLF`。为防止这种情况，请在上传前将行尾设置为 `LF`（换行）。*

Markdown 文件必须使用 `LF` 行尾序列。
